### Contenido Textual
1. Descripción General

1.1. Propósito

El objetivo de este documento es describir en profundidad el diseño de Agro App, una aplicación móvil destinada a la gestión integral del ganado bovino en los municipios michoacanos de Ocampo y Zitácuaro. La estructura del sistema, la descomposición en módulos, el diseño de las interfaces y el modelo de datos que orientarán la implementación del software se definen en este documento.

Este documento está dirigido a:

- Programadores: Para entender la estructura técnica del sistema y orientar su implementación
- Arquitectos de software: Con el fin de verificar las decisiones de diseño y los patrones empleados
- Evaluadores: Para comprobar que se cumplan los requerimientos fijados en el Análisis del Sistema
- Los mantenedores: Con el fin de simplificar las futuras ampliaciones y cambios del sistema

1.2. Objetivo del diseño

Los objetivos principales del diseño de Agro App.

- Completa funcionalidad sin conexión: Planear una estructura que asegure un funcionamiento completamente independiente de la conexión a internet, dándole prioridad al almacenamiento local y a la sincronización diferida.
- Facilidad de uso y simpleza: Diseñar una estructura modular que cuente con interfaces intuitivas adecuadas para personas con escasa alfabetización digital (nivel de educación primaria y secundaria).
- Óptimo desempeño: Garantizar que las operaciones críticas tengan tiempos de respuesta inferiores a 2 segundos, gracias a un diseño eficaz del almacenamiento en caché y de las consultas.
- Posibilidad de escalar en el futuro: Crear una base de arquitectura que posibilite la incorporación de capacidades avanzadas (como análisis predictivo, sincronización en la nube y multi-rancho) sin necesidad de una reestructuración significativa.

I. Información sobre la implementación del código fuente (nombres de métodos o clases concretas).

1.4.Definiciones, acrónimos y abreviaturas

**Términos de arquitectura:**

- **MVC (Model-View-Controller): Patrón de diseño de software que separa la lógica de negocio (Model), la interfaz de usuario (View) y el control de la interacción (Controller)**

- **MVVM (Model-View-ViewModel): Patrón de diseño que separa la lógica de negocio de la interfaz mediante un intermediario (ViewModel)**

- **DAO (Data Access Object): Patrón que proporciona una interfaz abstracta para acceder a la base de datos**

- **Singleton: Patrón que garantiza que una clase tenga una única instancia**

- **Repository Pattern: Patrón que media entre la capa de dominio y la capa de datos**

- **Offline-first: Arquitectura donde la aplicación funciona primariamente sin conexión a internet**

- **ACID: Propiedades de transacciones de base de datos (Atomicidad, Consistencia, Aislamiento, Durabilidad)**

**Acrónimos técnicos:**

- **API: Application Programming Interface**

- **CRUD: Create, Read, Update, Delete**

- **UI: User Interface**

- **UX: User Experience**

- **PK: Primary Key**

- **FK: Foreign Key**

- **SQL: Structured Query Language**

- **PDF: Portable Document Format**

- **ROI: Return On Investment**

- **SDK: Software Development Kit**

- **IDE: Integrated Development Environment**

- **RAM: Random Access Memory**

- **GB: Gigabytes**

**Término del dominio ganadero:**

- **Arete**: Dispositivo de identificación oficial colocado en la oreja del animal
- **Becerro/Ternero**: Cría de ganado bovino menor a 12 meses
- **Calendario sanitario**: Programa de vacunas y tratamientos preventivos
- **Desparasitante**: Medicamento para eliminar parásitos internos/externos
- **Forraje**: Alimento vegetal para el ganado (pastos, heno, alfalfa)
- **Hato**: Conjunto total de animales bovinos de un productor
- **Potrero**: Terreno cercado para pastoreo
- **Suplementación**: Aporte adicional de nutrientes a la dieta básica
- **Trazabilidad**: Sistema de identificación y seguimiento de animales
- **Carga animal**: Número de animales por hectárea de pastoreo
- **Garrapata**: Parásito externo que transmite enfermedades
- **Mastitis**: Inflamación de la glándula mamaria en vacas lecheras

**Componentes del sistema:**

- **Activity**: Componente de Android que representa una pantalla
- **Adapter**: Puente entre View y datos para listas
- **BroadcastReceiver**: Componente que responde a mensajes broadcast
- **SQLiteOpenHelper**: Clase para gestionar bases de datos SQLite
- **RecyclerView**: Vista para mostrar listas grandes eficientemente
- **AlertDialog**: Cuadro de diálogo para mensajes y confirmaciones
- **AlarmManager**: Servicio para programar tareas en segundo plano
- **CameraX**: Biblioteca para acceso a cámara del dispositivo
- **WorkManager**: API para tareas diferidas y periódicas

1. USUARIOS

- **Propósito:** Gestión de acceso a la aplicación
- **Registros Iniciales:** 1 (admin/admin123)
- **Reglas de Negocio:**
  - Username debe ser único
  - Password se almacena en texto plano.

2. ANIMALES (Entidad Central)

- **Propósito:** Registro maestro del ganado
- **Reglas de Negocio:**
  - numero_arete debe cumplir formato SINIGA (10 dígitos)
  - estado solo puede ser: Sano, Enfermo, Vendido, Muerto
  - Si estado = "Vendido", fecha_salida y precio_venta son obligatorios
  - Si estado = "Muerto", fecha_salida es obligatorio
  - fecha_nacimiento no puede ser futura
  - precio_compra por defecto es 0 (para animales nacidos en rancho)

3. CALENDARIO_SANITARIO

- **Propósito:** Programación y seguimiento de eventos sanitarios
- **Reglas de Negocio:**
  - Si recordatorio = 1, se programa AlarmManager
  - estado puede ser: Pendiente, Realizado, Cancelado
  - fecha_realizada solo se completa si estado = "Realizado"
  - Notificación se programan 3 días antes, 1 día antes y ese mismo día de cadaevento programado. hora_recordatorio

4. HISTORIAL_CLINICO

- **Propósito:** Bitácora médica del animal
- **Reglas de Negocio:**
  - Estado puede ser: En Tratamiento, Recuperado, Crónico
  - Cada registro representa un episodio de enfermedad
  - Permite múltiples registros por animal

## SECCIÓN: Página 16 - Continuación Descripción Entidades

### Contenido Textual
5. GASTOS

- **Propósito:** Control financiero de inversiones
- **Reglas de Negocio:**
  - tipo puede ser: Alimento, Medicamento, Vacuna, Equipo, Mano de Obra, Infraestructura, Otro
  - monto siempre debe ser positivo
  - Se suma automáticamente a la inversión total del animal

6. ALIMENTACION

- **Propósito:** Registro nutricional del ganado
- **Reglas de Negocio:**
  - tipo_alimento puede ser: Pasto, Forraje, Concentrado, Grano, Suplemento, Otro
  - unidad puede ser: kg, g, toneladas, pacas, litros
  - cantidad debe ser positiva
  - Costo es opcional

**Relaciones y Cardinalidades:**

| Relación | Cardinalidad | Eliminación en Cascada |
|---|---|---|
| USUARIOS → ANIMALES | 1: N | No (v1.0) |
| ANIMALES → CALENDARIO_SANITARIO | 1: N | Sí (ON DELETE CASCADE) |
| ANIMALES → HISTORIAL_CLINICO | 1: N | Sí (ON DELETE CASCADE) |
| ANIMALES → GASTOS | 1: N | Sí (ON DELETE CASCADE) |
| ANIMALES → ALIMENTACION | 1: N | Sí (ON DELETE CASCADE) |

Justificación de **ON DELETE CASCADE:** Cuando se elimina un animal, todos sus registros relacionados (eventos sanitarios, historial clínico, gastos y alimentación) deben eliminarse automáticamente para mantener la integridad referencial y evitar datos huérfanos.

4.3. Reglas de Integridad

Reglas de Negocio:

1. **Formato Arete**: Debe cumplir con estándar SINIGA (10 dígitos numéricos)

2. **Estados Finales**: Animales en estado "Vendido" o "Muerto" no permiten modificaciones

3. **Fechas Válidas**: fecha nacimiento no puede ser futura; fecha_salida debe ser posterior a fecha_ingreso

4. **Precios Positivos**: precio_compra, precio_venta, monto deben ser \(\geq 0\)

5. **Cardinalidades**: Un animal puede tener múltiples eventos sanitarios, pero un evento sanitario pertenece a un solo animal (opcional)

Restricciones de Base de Datos:

- **Claves Primarias**: Todas las tablas tienen ID autoincremental
- **Claves Foráneas**: Relaciones con eliminación en cascada para datos dependientes
- **Unicidad**: numero_arete y username deben ser únicos
- **Valores No Nulos**: Campos críticos no pueden ser NULL

Reglas de Cálculo:

- **Inversión Total**: Suma de precio_compra + gastos asociados al animal
- **ROI**: (precio_venta - inversión_total) / inversión_total * 100
- **Edad Animal**: Diferencia en meses entre fecha_actual y fecha_nacimiento
